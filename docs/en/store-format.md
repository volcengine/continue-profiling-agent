# CPA Store Format

[Back to README](../../README.md)

## Scope

CPA currently produces two kinds of output artifacts:

- a continuous store directory written by regular `cpa monitor`
- a one-shot folded flamegraph file written by `cpa monitor --oneshot`

This page focuses on the first kind: the on-disk directory format consumed by
`cpa show` and the embedded Rust `cpa_show` TUI.

## Directory Layout

A typical continuous store looks like:

```text
cpa_YYMMDD/
├── conf
├── strmap
├── idsmap
├── stack.bin
└── decompressed/        # cache generated by cpa show
```

Notes:

- `conf` is plain text.
- `strmap`, `idsmap`, and `stack.bin` are zstd-compressed.
- `decompressed/` is not part of the producer-side wire format. It is a viewer
  cache created on demand by `cpa show`.

## `conf`

`conf` is written when `cpa monitor` creates a new store directory. It uses a
simple `key: value` text format:

```text
version: 1.0.0
cpu_num: 128
freq: 49
record_interval: 1
persistent_day: 7
record_env_name: POD_NAME
kernel_stack: 0
stack_size: 8192
```

The current implementation writes these keys:

- `version`
- `cpu_num`
- `freq`
- `record_interval`
- `persistent_day`
- `record_env_name`
- `kernel_stack`
- `stack_size`

Readers ignore unknown keys in `conf` and continue with supported keys.

## `strmap`

`strmap` is the string table. Each line is:

```text
<string> <id>\n
```

Example:

```text
java;main;foo 123
bar_[k] 124
abc            xyz            -67890-3-1234|pod-a 125
```

Usage:

- regular stack frames are interned into `strmap`
- the metadata header is also stored as a string and becomes the first string id
  of a stack
- kernel frames are typically distinguished by the `_[k]` suffix

`strmap` is append-only within one store directory.

## `idsmap`

`idsmap` maps one folded stack to the list of string ids that form it. Each
line is:

```text
<sid1>;<sid2>;...; <ids_id>\n
```

Example:

```text
125;124;123; 10
```

Meaning:

- `ids_id` is the logical identifier for one aggregated stack
- the left-hand side is the ordered list of string ids
- the first `sid` is conventionally the metadata entry
- subsequent `sid` values are the actual frame sequence

Like `strmap`, `idsmap` is appended over time.

## `stack.bin`

`stack.bin` is the core time-series file. It is zstd-compressed, but its
decompressed payload is binary.

### Top-Level Layout

Each record uses the following layout. Integers are little-endian:

```text
u8[2]   header      = 0xFA 0xFB
u64     start_ms
u64     end_ms
u64     entry_len
repeat entry_len times:
    u32 ids_id
    u64 count
u8[2]   footer      = 0xFC 0xFD
```

### Field Semantics

- `start_ms` / `end_ms`
  - millisecond offset from local day start (`00:00:00`)
  - readers adjust cross-day windows by adding one day when `end_ms` falls
    behind the first record start
- `entry_len`
  - number of aggregated items in the record
- `ids_id`
  - logical stack id pointing into `idsmap`
- `count`
  - aggregated sample count for that stack in this record window

### Minimal Example

```text
[0xFA,0xFB]
start_ms=1000
end_ms=2000
entry_len=2
  (ids_id=10, count=100)
  (ids_id=11, count=200)
[0xFC,0xFD]
```

### Reader-Side Notes

- the C-side `cpa show` scans records by validating header and footer pairs
- the Rust `cpa_show` loader currently drops the final indexed record as a
  defensive measure because the last append may still be in flight
- because the file is append-oriented, production readers should tolerate a
  partially flushed tail

## Metadata Encoding

The first `sid` of each aggregated stack points to a metadata string. The
current serialized form is:

```text
<comm(15 bytes, space padded)+group_comm(15 bytes, space padded)>-<cgroup_id>-<cpu>-<pid>|<env>
```

Example:

```text
nginx          nginx          -123456-7-4321|pod-a
```

Field meanings:

- `comm`
  - thread name, fixed-width 15 bytes
- `group_comm`
  - process-group / group-leader comm, fixed-width 15 bytes
- `cgroup_id`
  - cgroup id collected from the sampled task
- `cpu`
  - sampled CPU
- `pid`
  - process pid
- `env`
  - value selected from configured `record_env_name` keys; falls back to
    `SYSTEM` when nothing matches

## Hidden Dependencies and Accuracy Notes

### `env` / `pod`

The Rust TUI and filters often expose the `env` field as `pod`, but the field
still comes from recorded environment variables and does not imply direct
integration with the Kubernetes API.

The actual implementation is:

- `cpa monitor --record_env_name` configures one or more environment variable
  keys
- the runtime scans `/proc/<pid>/environ` in order and keeps the first matching
  key's value
- when no configured key exists, metadata stores `SYSTEM`

This means:

- if your workload does not inject `POD_NAME`, `SERVICE`, or similar values
  into its environment, the `pod` dimension is not authoritative
- when multiple candidate keys exist, only the first match is persisted
- the value is cached per pid context, so it can lag behind a later runtime
  environment change

### `cgroup_id`

`cgroup_id` is much harder than `env/pod` on the BPF path:

- it comes directly from the kernel-side sampled task
- it does not depend on user-space conventions
- it does not depend on environment injection

One caveat: the current perf backend does not populate cgroup id the same way,
so perf-generated stores may carry `0` in that column.

## Design Tradeoffs

### Why persist `pid` but not `tid`

The store persists `pid` in metadata, but not `tid` as a primary long-lived
dimension. That is intentional for continuous profiling:

- `tid` has much higher cardinality and shorter lifetime
- persisting it directly would inflate `strmap` and `idsmap`
- thread pools and short-lived worker threads would make longitudinal diffs
  noisy and unstable

The compromise is:

- metadata still carries `comm` and `group_comm`
- the viewer can surface thread names with `--show_thread_name`
- long-term aggregation stays centered on more stable labels such as
  `pid/group_comm/cgroup_id/env`

### Why use a directory format instead of an online protocol

CPA currently prefers `store directory + offline reader` over a live RPC:

- `cpa show` and the embedded TUI consume the same store
- incident handling is easier because a store can be archived and replayed
- the binary `stack.bin` plus text sidecars work well with append-heavy writes

This is also why `decompressed/` is treated as a viewer cache, not as part of
the producer contract.
